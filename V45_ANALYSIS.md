## 架构分析

### 继承关系
```
OTEHV2RankEvent → OTEventHazardV2Survival (31_ot_event_hazard_v2/model_v2.py)
```
- **没有脱离基线框架**：保持OT诱导的事件风险分解范式
- **预测路径**：避免SlotDecoder/CrossAttention/SelfAttention/D×3拼接
- **骨干结构**：与v9相同的OT骨干 + capacity tuning (num_events=24, heads=4, layers=4)

### 核心创新点（相比v9）
1. **Per-event生存监督**：监督每个事件的logits，不仅仅是平均事件logits
2. **Cox风格成对排序损失**：基于风险得分的排序监督
3. **全局事件残差头**：额外的全局分类头，与事件平均logits加权求和
4. **OT epsilon退火**：从eps_start=0.10逐渐退火到eps_end=0.05（12个epoch）
5. **门控熵正则化**：防止门控分布过于尖锐，促进多样性

### 参数配置
```python
self.lambda_per_event = 0.15      # 每个事件的NLL损失权重
self.lambda_rank = 0.15           # 排序损失权重
self.lambda_global_cons = 0.02    # 全局一致性损失权重
self.lambda_gate_ent = 0.005      # 门控熵正则化权重
self.rank_margin = 0.0            # 排序损失边界
self.rank_max_pairs = 4096        # 最大排序对数量
```

## V45-V49消融实验关键发现

| 变体 | C-index | 结论 |
|------|---------|------|
| **V45 全开** | **0.7105** | **基准（SOTA）** |
| V46 仅 per-event NLL | 0.6977 | 单加event NLL**拖累**（相比V45 -0.0128） |
| V47 仅 ranking loss | 0.7050 | 单加ranking**不够**（相比V45 -0.0055） |
| V48 仅 global residual | 0.6947 | 单加global**拖累**（相比V45 -0.0158） |

### 关键洞察
1. **协同效应**：4个改进必须**同时使用**才有效
2. **排序损失是核心**：V47 (仅ranking) 0.7050 > V46 (仅NLL) 0.6977 > V48 (仅global) 0.6947
3. **ε退火是催化剂**：V49显示退火有帮助，但单独使用不如与ranking结合
4. **全局残差需要谨慎**：单独使用全局残差会拖累性能

## V44失败教训分析

### V44 (otehv2_boost) 错误方向
1. **叠加过多技巧**：uncertainty weighting + transport consistency + IPCW + drop-path
2. **破坏协同**：V45的4个改进是调谐好的协同集，新增技巧打破平衡
3. **数值不稳定**：uncertainty weight + IPCW的双重权重导致训练不稳定
4. **过拟合**：4个fold在epoch 14-15触发早停，fold 3严重崩坏(0.6224)

### 正确改进原则
1. **一次只改一处**：基于V45，每次只添加一个独立改进
2. **保持协同集完整**：保持V45的4个核心改进不变
3. **验证充分**：新技巧必须通过消融实验验证有效性
4. **逐步扩展**：成功后再考虑组合多个已验证的技巧

## 风险与应对

### 主要风险
1. **过拟合**：新技巧可能导致过拟合
2. **训练不稳定**：多技巧叠加导致梯度爆炸
3. **计算成本**：实验数量增加导致时间成本上升

### 应对策略
1. **严格验证**：每个改进必须通过5-fold验证
2. **渐进式添加**：一次只添加一个验证有效的技巧
3. **早期监控**：实时监控训练曲线，及时终止失败实验
4. **资源优化**：优先运行高价值实验，并行化低优先级实验
